## 1. Фиксированный сценарий сравнения

> *Экран списка задач* должен выводить поля `id`, `title`, `done`.  
> *Экран детального просмотра* должен выводить поля `id`, `title`, `description`, `done`.  
> Дополнительное действие — создание новой задачи (и, опционально, обновление статуса `done`).

## Итоговая таблица сравнения


| Критерий | REST | GraphQL |
|----------|----------|----------|
| Структура API    | Несколько endpoint’ов под ресурсы   | Единая точка входа /graphql   |
| Выбор полей    | Определяет сервер (фиксированный JSON)   | Определяет клиент   |
| Избыточность данных    | Возможна (over‑fetching)   | Минимальна   |
| Обработка ошибок    | HTTP‑статусы + тело   | HTTP 200 + поле errors   |
| Кэширование   | Простое, по URL   | Сложное, требует дополнительных решений  |
| Простота внедрения    | Высокая   | Ниже (схема, резолверы)   |
| Инструменты тестирования   | curl, Postman, Swagger  | GraphiQL, Apollo Studio, curl (POST)  |


## Вывод

Для реализованного сценария — классического CRUD-сервиса 
с ограниченным числом экранов — REST оказался более простым и быстрым в реализации. 
Он не требует отдельной схемы, ошибки прозрачны через HTTP‑статусы, а незначительный 
over‑fetching (одно лишнее поле description) не создаёт серьёзной нагрузки.
GraphQL показал преимущество в точной выборке полей и удобной самодокументируемости. 
Для простых CRUD-проектов REST остаётся практичнее и легче в сопровождении.

## КВ

**Принципиальное отличие REST и GraphQL?**


REST — архитектурный стиль с множеством URL‑адресов и фиксированными форматами ответов, GraphQL — язык запросов, где клиент сам формирует структуру ответа через единую точку входа.

**Что такое over‑fetching и under‑fetching?**

Over‑fetching — получение лишних данных, не нужных клиенту. Under‑fetching — недостаточность одного запроса, вынуждающая делать дополнительные.

**Почему GraphQL позволяет клиенту точнее выбирать поля ответа?**

Клиент явно перечисляет требуемые поля в запросе, сервер возвращает только их.

**Почему REST проще кэшировать стандартными средствами HTTP?**

У каждого ресурса уникальный URL, что позволяет кэширующим прокси и браузерам сохранять ответы по ключу URL.

**Чем отличается обработка ошибок в REST и GraphQL?**

REST использует коды состояния HTTP (404, 400, 500), GraphQL — возвращает 200 и помещает ошибки в массив errors.

**В каких случаях REST оказывается более практичным решением?**

Для простых CRUD-сервисов, стабильных ответов и ограниченного числа клиентов.

**В каких случаях GraphQL может дать преимущества?**

При множестве клиентских приложений с разными требованиями к данным, сложных графах объектов и необходимости избежать каскадных запросов.

**Почему корректное сравнение нужно проводить на одном и том же сценарии?**

Иначе сравниваются разные задачи, а не сами подходы, что делает вывод необъективным.

**Какие сложности возникают при сопровождении GraphQL API?**

Поддержка схемы и резолверов, риск N+1 проблемы, сложности кэширования и мониторинга.

**Почему для учебных CRUD‑сервисов REST часто оказывается проще?**

REST интуитивен, не требует изучения языка запросов, быстрее реализуется и тестируется стандартными инструментами.
